Blocks, Chains & Forks

What's actually inside a block, why stacking them into a chain makes history expensive to rewrite, and what happens when two blocks tie for the same spot.

What's in a block

A header, and the data it vouches for

Every block has two parts: a small header that's cheap to check, and a body full of transactions that the header vouches for.

Header

  • Hash of the previous block — this is what makes it a chain
  • A Merkle root summarizing every transaction inside (see Merkle Trees)
  • Timestamp
  • Nonce / proof data the consensus mechanism needs

Body

  • The full list of transactions this block includes
  • Nothing here is trusted directly — it's only valid if it hashes into the Merkle root the header claims

Notice the header doesn't store the transactions themselves — just a hash that summarizes them. That's the Merkle root doing exactly the job described in the last topic.

Why a chain, not just a pile of blocks

Each block locks in everything before it

Because every block's header stores the hash of the block before it, the blocks form a chain of commitments. Changing anything in an old block changes that block's hash — which no longer matches what the next block claims as its "previous hash." The break is immediately visible, and fixing it means recomputing every block after the change, not just the one you edited.

Try it: edit any block's data below and watch the chain break right after it. Then re-mine forward to fix it — notice how much work that takes the deeper the edit is.

When two blocks tie

Forks: the network briefly disagrees

Sometimes two miners or validators produce a valid next block at almost the same moment. For a short while, different nodes see different "tips" of the chain — a fork. Most networks resolve this with a simple rule: whichever branch ends up with more accumulated work (or more stake, more votes — depending on the consensus mechanism) wins. Blocks on the losing branch are orphaned: discarded, with their transactions returned to the pool to be included later.

These natural, temporary forks resolve themselves within a block or two. Hard forks and soft forks are a different thing entirely — deliberate protocol changes. A hard fork changes the rules in a way old software can't validate, permanently splitting the chain in two if not everyone upgrades (Bitcoin / Bitcoin Cash, Ethereum / Ethereum Classic). A soft fork tightens the rules in a backward-compatible way, so old nodes keep accepting the new blocks without even needing to upgrade.

Where this shows up

Confirmations exist because of this

  • "Wait for 6 confirmations" on Bitcoin is just waiting for 6 blocks to pile on top of yours, so an attacker would need to out-mine the entire network to ever orphan it.
  • Proof-of-stake chains with finality (much of modern Ethereum, Cosmos) avoid long fork uncertainty by having validators explicitly vote a block as final — after that vote, it's a hard guarantee, not a probability.
  • This is the same CAP-theorem tension from two topics ago: a chain that's briefly forked is choosing availability over instant consistency, and resolving the disagreement later.
