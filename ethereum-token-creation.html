How Ethereum actually works

There's no token printer. There's only code that says how many to make.

Ethereum doesn't decide how many units of a token can exist, who's allowed to create one, or whether there's a ceiling. Every one of those rules is written into a smart contract by whoever deploys it. Flip the switch below to see the difference a single line of code makes.

No gatekeeper

Anyone can deploy a contract — that's the point

Ethereum is permissionless: there's no company or committee that approves a new token. In practice, four kinds of people end up doing it.

Solo

Individual developers

Anyone who can write and deploy a smart contract can mint a token — there's no licence required, just code and a bit of ETH for gas.

Startups

Projects launching a token

New apps and protocols often issue their own currency, governance token, or in-app point system as part of launching their product.

Platforms

Issuers & tokenization platforms

Companies that tokenize a real-world asset — a fund, a building, a painting — deploy the contract on the issuer's behalf and manage it afterward.

Protocols

DAOs & automated contracts

Some tokens are minted automatically by other contracts — like a lending protocol issuing a receipt token the moment you deposit collateral.

Pick a shape, not just a number

Three standards cover almost every token

The standard a developer picks decides whether units are identical or unique.

ERC-20

Fungible

Every unit is identical and interchangeable — one token is worth exactly the same as any other of its kind.

e.g. stablecoins, governance tokens, in-app points
ERC-721

Nonfungible

Each token is unique, with its own ID and attributes — no two are interchangeable, even within the same contract.

e.g. one-of-a-kind collectibles, deeds, memberships
ERC-1155

Both, in one contract

A single contract can manage many token types at once — some fungible, some unique — which saves on deployment cost.

e.g. game items with both currency and unique gear
Back to the question

So — is there a hard cap?

Only if the contract's author wrote one in. In practice, most tokens fall into one of three patterns.

Fixed supply

Hard-capped

The contract sets a maximum once, and mint() simply refuses to create more past that number — no matter who calls it.

Continuous

Open-ended

No ceiling at all — new tokens mint whenever a condition is met. Stablecoins work this way: more get minted as more deposits come in, and burned on redemption.

Asset-pegged

Capped to a backing asset

supply ≡ value backed

The cap is set to match a real-world asset's value — e.g. a $1M building split into exactly 1,000 tokens. Minting beyond that would create tokens with nothing behind them.

A limit that never matters

There is one true ceiling — it's just never the real constraint

Token balances are usually stored as a uint256 — a number that can hold values up to roughly 1.15 × 10⁷⁷. Whatever cap actually applies to a token, it's the one its author chose to write in, not anything Ethereum itself enforces.

The other kind of limit

Even uncapped tokens aren't free to mint

There's no protocol-level supply limit, but every action still costs something — and that cost is its own kind of brake.

Paid in ETH

Deploying a contract and calling its mint function both require a gas fee, paid in Ethereum's native currency, ETH.

Scales with demand

Gas prices rise and fall with how congested the network is — minting during a busy period costs more than minting on a quiet day.

A built-in deterrent

Even an "open-ended" contract with no coded cap isn't free to spam — every extra mint still has to be paid for, which naturally discourages unlimited minting.
