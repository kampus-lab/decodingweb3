Cryptographic Hash Functions

The one-way function that everything else in this course quietly depends on — Merkle trees, block linking, mining, addresses, all of it.

What it is

Any input, one fixed-size fingerprint

A cryptographic hash function takes data of any size — a single letter or an entire movie file — and produces a fixed-length string of bits called a digest. Feed it the same input twice, you get the same digest. Change one character, and the digest comes out completely, unpredictably different.

Deterministic

The same input always produces the exact same digest — there's no randomness involved.

Fixed-size output

A single word and an entire book both come out the same length — 256 bits for SHA-256.

Avalanche effect

Changing even one character flips roughly half the output bits, with no visible pattern.

One-way

Going from input to digest is fast. Going from digest back to input is, by design, computationally infeasible.

Notice both digests are always 64 hex characters (256 bits) — no matter how long or short the input is — and that a one-character difference shares roughly no characters in common with the original. That's the avalanche effect, not a coincidence.

This is exactly the building block behind the Merkle tree from an earlier topic — every node in that tree was just this function, called over and over. It's also what Proof of Work mining actually searches for: a hash function is run billions of times per second, hunting for one input (a "nonce") whose digest happens to start with enough zeros.

Where this shows up

Quietly everywhere

  • Block hashes & Merkle roots — every block and every transaction tree is identified by a hash, which is exactly why tampering is detectable.
  • Git commit IDs — every commit hash is a SHA-1 (now migrating to SHA-256) digest of its contents and parent commit, which is why a single changed line gives a totally different commit hash.
  • Password storage — sites store a hash of your password, never the password itself, so a stolen database doesn't directly hand over your login (though real systems also need salting, which is its own topic).
  • File integrity checks — software downloads are often published alongside a hash so you can verify the file wasn't corrupted or tampered with in transit.
