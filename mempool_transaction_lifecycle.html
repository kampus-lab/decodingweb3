Mempool & Transaction Lifecycle: From Signature to Settled

Signing a transaction doesn't put it on the chain — it puts it in an unsynced waiting room first. Send one through yourself and watch every stage in between.

The journey

Six stages, one transaction

A transaction does not jump straight onto the chain the moment you sign it. It passes through a short pipeline, and at any stage along the way it can stall, get bumped by something with a higher fee, or simply wait its turn.

Sign transaction→ Broadcast→ Mempool→ Validator selects→ New block→ Confirmed

Worth remembering: none of this is instant or guaranteed until the last step. A signed transaction sitting in a mempool can still be replaced or dropped.

What is a mempool

A waiting room with no single copy

Mempool — memory pool — is the set of transactions a node has heard about but that haven't been confirmed into a block yet. Every full node keeps its own copy in memory, built only from whatever it has personally received and verified so far.

That copy is not identical everywhere. Propagation takes time, so two nodes can briefly disagree about exactly which transactions are pending — one might have already heard about a transaction the other hasn't seen yet.

When it gets stuck

Stuck transactions and replace-by-fee

If your fee is too low relative to everyone else waiting, validators simply keep picking other transactions ahead of you, block after block. Your transaction is not failed — it is just losing every auction.

Most wallets let you fix this with replace-by-fee (RBF): you resubmit the same transaction with the same nonce but a higher fee, telling the network the new version should be considered instead of the old one. Sending a replacement with the original amount set to zero is the common way to effectively cancel a stuck transaction.

Why your node sees something different

There is no global mempool

Because every node's view is independently assembled, two wallets — or a wallet and a block explorer — can briefly show conflicting "pending" states for the same transaction. Mempools also have a size limit: when they fill up, nodes evict the lowest-fee entries to make room, which is part of why chronically underpriced transactions eventually vanish rather than waiting forever.
