Full, Light & Archive Nodes: Who Keeps What, and Why

Three answers to the same question — how much of the chain's history are you willing to store, and what will you trust someone else for instead?

The trade-off

Storage, trust, and convenience

Every node on the network is answering the same questions the others are, but how much history it stores — and how much it trusts someone else for the rest — is a real design choice with real storage and bandwidth costs attached. Full, archive, and light nodes are three different bets on that trade-off.

Full node

Verifies everything, remembers mostly the present

A full node downloads every block and independently checks every transaction against the consensus rules — nobody has to take anyone's word for it. What it keeps afterward is more selective: it holds the entire block history, but by default it prunes older snapshots of account and contract state, keeping only what the chain looks like right now. Running one costs hundreds of gigabytes of storage, and it is the thing that actually makes the network trustless — anyone can run one and check for themselves.

Archive node

Never forgets a single block

An archive node is a full node that refuses to throw anything away. Alongside the full block history, it keeps the complete state at every single block height, so a query like "what was this address's balance at block 4,000,000" gets an instant, exact answer instead of a shrug. That historical queryability is exactly what block explorers, RPC providers, and analytics platforms are built on — at the cost of multiple terabytes of storage and a much longer initial sync.

Light node

Trusts proofs instead of storing data

A light client does not store the chain or validate transactions at all. It keeps block headers only, and asks a full node for a specific cryptographic proof — "show me that this transaction is really included in this block" — rather than re-executing everything itself. That is how a mobile wallet or browser extension can work from a few kilobytes instead of hundreds of gigabytes. Bitcoin's version of the same idea predates Ethereum's by years, under the name SPV: simplified payment verification.

Worth remembering: a light node's security depends entirely on the honesty and availability of whichever full node it's asking — it has opted out of verifying, not just storing.
